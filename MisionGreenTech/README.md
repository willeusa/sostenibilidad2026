# Caso Práctico GreenTech

## Auditoría, Refactorización y Defensa de un MVP Sostenible

**Modalidad:** Individual

# 1\. El Escenario: Tu primer día como Consultor

Ponte en situación. Has sido contratado como Especialista en Sostenibilidad TIC por la agencia **GreenTech Solutions**. Tu primer encargo parece sencillo: auditar y lanzar el Producto Mínimo Viable[^1] (MVP) de una landing page para una importante ONG internacional que lucha contra el cambio climático.

El problema es que el becario anterior dejó un código que es una auténtica contradicción. La web, cuyo mensaje es "salvar el planeta", tarda horrores en cargar, devora la tarifa de datos de los usuarios y sobrecalienta los procesadores de los teléfonos de gama baja. Fíjate en la ironía: por culpa de importar un framework mastodóntico como Bootstrap entero solo para darle color verde a un botón, estamos fomentando que los usuarios necesiten móviles más potentes, acelerando la obsolescencia programada y generando basura electrónica (e-waste).

# 2\. Tu Misión

Como profesional del desarrollo, no vale con decir "esto está mal". Tienes que solucionarlo, documentarlo y demostrar que tu alternativa funciona. Para ello, deberás:

1. **Auditar y Refactorizar (El Código)**. Analiza el index.html original. Elimina toda la "grasa digital" (dependencias inútiles, *frameworks* pesados). Sustitúyelo por CSS nativo, implementa carga perezosa (*lazy loading*) para las imágenes y optimiza las fuentes. Queremos una web que no pese, pero que visualmente cumpla con los requisitos del cliente.  
2. **Repositorio en GitHub (El MVP)**. Nada de enviar archivos comprimidos por correo; somos profesionales. Crearás un repositorio público en GitHub llamado GreenTech-MVP-Sostenible\_Version\_TusNombres\_Apellidos.  
   1. Debe contener el código final refactorizado.  
   2. Se valorará un historial de *commits* semántico y lógico (ej. refactor: elimina bootstrap y añade css nativo).  
   3. El README.md funcionará como tu **Informe Ejecutivo**. Aquí debes explicar (apoyándote en fuentes académicas) cómo tú reducción de ancho de banda disminuye el carbono embebido y alarga la vida útil del hardware.  
3. **La Defensa ante el Cliente (Roleplay)**. El próximo día de clase, tendrás un **máximo de 5 minutos** para presentar tu MVP a la directiva de GreenTech Solutions (tu equipo docente). Tendrás que justificar tus decisiones técnicas, explicar cuánto peso has ahorrado y convencer al cliente de por qué tu código ahora sí es verdaderamente responsable con el medio ambiente.

# 3\. Criterios de Evaluación (Lo que el cliente espera de ti)

1. **Excelencia Técnica y Profesionalidad (35%)**. El repositorio de GitHub está inmaculado. El uso de HTML/CSS nativo es impecable, ligero y escalable. Las imágenes y recursos están optimizados al máximo.  
2. **Visión de Economía Circular y ASG (35%)**. El *README.md* documenta perfectamente el impacto físico del código. Logras relacionar la ligereza del software con el ahorro de energía en los centros de datos y la prevención de la obsolescencia del dispositivo del usuario final.  
3. **Defensa y Habilidades Comunicativas (30%)**. Eres capaz de sostener la mirada al cliente. Respondes con aplomo a las preguntas técnicas y argumentas el valor añadido que la sostenibilidad aporta al producto, sin titubear.

# 5\. Marco Normativo y de Evaluación (RA y CE)

Para que sepas exactamente qué habilidades oficiales estamos midiendo, esta actividad se enmarca dentro de la programación del módulo, evaluando de forma directa los siguientes apartados:

* **Resultado de Aprendizaje (RA)**  
  * **RA4**. Aplica criterios de economía circular y ecodiseño en las fases del ciclo de vida de los productos y servicios propios del sector profesional, proponiendo soluciones que minimicen el impacto ambiental.  
* **Criterios de Evaluación (CE) involucrados**  
  * **CE 4.a)** Se han identificado los principios de la economía circular y su aplicación al ciclo de vida del producto/servicio (en nuestro caso, el impacto del software en el hardware).  
  * **CE 4.c)** Se han aplicado técnicas de ecodiseño en la propuesta de productos o servicios (refactorización, eliminación de bloatware y optimización de carga).  
  * **CE 4.e)** Se ha valorado cómo las decisiones de diseño y desarrollo influyen en la prevención de la obsolescencia programada y la generación de residuos.

# 6\. Formato y Metodología de Entrega

Como hemos comentado, no buscamos una entrega tradicional, sino una dinámica profesional. Los pasos para entregar tu MVP son los siguientes:

1. **Enlace al Repositorio**. Deberás subir la URL pública de tu repositorio de GitHub a la tarea habilitada en Classroom.  
2. **Copia de Seguridad (Opcional pero recomendada)**. Classroom te permitirá adjuntar un archivo .zip con tu proyecto local por si el día de la presentación tenemos alguna caída de red o problemas con los servidores de GitHub.  
3. **Fecha Límite**. El enlace debe estar subido a Classroom en la fecha que se indica. Los *commits* realizados después de esa hora no se tendrán en cuenta para la evaluación técnica.


[^1]:  Un Producto Mínimo Viable (MVP) es la versión más básica y funcional de un nuevo producto, que incluye solo las características esenciales para satisfacer a los primeros usuarios (*early adopters*) y validar una idea de negocio en el mercado con el mínimo esfuerzo y coste posible.
